@moleculer/channels
Reliable durable messaging for Moleculer services
View Source ↗ Upstream

Unlike Moleculer's built-in emit which is fire-and-forget, @moleculer/channels is a persistent, durable, and reliable message delivery solution. Messages are stored in an external queue/stream server and only removed after successful acknowledgement. Supports consumer groups, dead letter queues, retry with backoff, and max-in-flight control.

Supported Adapters
Redis Streams ✓ RabbitMQ / AMQP ✓ NATS JetStream ✓ Apache Kafka ✓ In-memory (Fake) ✓
✅
ACK/NACKExplicit acknowledgement — messages stay in queue until ACK'd
🔁
Retry + BackoffAutomatic retry with exponential delay on NACK
💀
Dead Letter QueueFailed messages routed to DLQ after max retries
👥
Consumer GroupsMultiple service instances share load within a group
🚦
Max In-FlightBackpressure control per consumer
🔌
External IntegrationReceive messages from non-Moleculer services
channels — JavaScript usage 
// moleculer.config.js
const ChannelsMiddleware = require("@moleculer/channels").Middleware;

module.exports = {
  middlewares: [
    ChannelsMiddleware({ adapter: "redis://localhost:6379" })
  ]
};

// payments.service.js — subscribe to durable channel
module.exports = {
  name: "payments",
  channels: {
    async "order.created"(payload) {
      await processOrder(payload);
      // auto-ACK on return, or call this.ack(msg) manually
    }
  }
};

🦀 Rust Implementation Status

moleculer-rs includes a full in-memory channels implementation: ChannelDef, ChannelMessage, ACK/NACK, consumer groups, DLQ, and exponential backoff retry. External adapters (Redis Streams, AMQP, NATS, Kafka) are on the roadmap — contributions welcome.

@moleculer/database
Advanced database access mixin for Moleculer services
View Source ↗ Upstream

A powerful database service mixin that follows the one database per service pattern. Provides CRUD actions, pagination, field filtering, validations, and data transformations out of the box. Supports multiple adapters including NeDB (in-process), MongoDB, and SQL databases via Knex.

Supported Adapters
NeDB (in-process) ✓ MongoDB ✓ Knex (MySQL, PostgreSQL, SQLite) ✓
⚡
CRUD ActionsAuto-generated list, get, create, update, remove actions
📄

🔍
Field FilteringRequest only the fields you need per query
✅
ValidationSchema-based input validation and sanitization
🔒
Field PermissionsRead-only, write-only, and virtual field support
🧩
PopulateCross-service entity population (like SQL JOINs)
🗑️
Soft DeleteMark records deleted without removing from DB
🏢
Multi-tenancyBuilt-in scope isolation per tenant
database — service definition 
const DbService = require("@moleculer/database").Service;

module.exports = {
  name: "posts",
  mixins: [DbService({ adapter: "NeDB" })],
  settings: {
    fields: {
      id:        { type: "string", primaryKey: true, columnName: "_id" },
      title:     { type: "string", max: 255, required: true },
      content:   { type: "string" },
      votes:     { type: "number", integer: true, min: 0, default: 0 },
      createdAt: { type: "number", readonly: true, onCreate: () => Date.now() },
    }
  }
};

// Auto-generated actions: posts.list, posts.get,
// posts.create, posts.update, posts.remove

🦀 Rust Roadmap

A DatabaseMixin trait for moleculer-rs is planned — providing the same CRUD action scaffolding with adapters for SQLx (PostgreSQL, MySQL, SQLite), MongoDB, and SurrealDB. Field schemas map naturally to Rust structs with serde derive macros.

@moleculer/workflows
Reliable & scalable workflow orchestration (like Temporal.io)
View Source ↗ Upstream

Reliable, scalable workflow management for Moleculer — inspired by Temporal.io and Restate. Define multi-step workflows as regular service methods; the framework handles persistence, retries, concurrency control, timeouts, and event-driven state transitions. Workflows survive service restarts and crashes.

Supported Adapters
Redis ✓ Fake/In-memory ✓ More planned
💾
Durable ExecutionWorkflow state persisted — survives restarts
🔁
Retry PoliciesConfigurable retry strategies per workflow step
⏱️
TimeoutsPer-workflow and per-step timeout control
📡
SignalsSend events into running workflow instances
⚖️
ConcurrencyLimit parallel executions per workflow type
📊
Metrics & HistoryBuilt-in monitoring with configurable retention
workflows — user sign-up example 
const WorkflowsMiddleware = require("@moleculer/workflows").Middleware;

broker.createService({
  name: "users",
  workflows: {
    signupWorkflow: {
      timeout:     "1 day",
      retention:   "3 days",
      concurrency: 5,
      params: { email: { type: "email" } },
      async handler(ctx) {
        await ctx.call("mail.sendVerification", ctx.params);
        await ctx.waitForSignal("email.verified", "24h");
        await ctx.call("accounts.activate", ctx.params);
        return { success: true };
      }
    }
  }
});

// Start the workflow
const result = await broker.wf.run("users.signupWorkflow", { email: "me@example.com" });

🦀 Rust Roadmap

moleculer-rs plans a WorkflowDef API mirroring this design — durable async state machines backed by tokio tasks with Redis persistence. The waitForSignal pattern maps naturally to Tokio's async oneshot channels combined with persistent state snapshots.
